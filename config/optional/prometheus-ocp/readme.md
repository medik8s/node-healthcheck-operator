# Platform Prometheus Metrics Configuration

This configuration enables metrics collection via OpenShift Platform Prometheus (prometheus-k8s in openshift-monitoring namespace) using TLS client certificate authentication.

## Requirements

- OpenShift Container Platform 4.14+
- The `scrapeClass: tls-client-certificate-auth` feature requires OCP 4.14 or later

## How It Works

The Node Health Check Operator exposes metrics via kube-rbac-proxy, which is configured to:

1. **Serve TLS**: Using a certificate generated by OpenShift's service-serving-cert-signer
2. **Verify Client Certificates**: Using the CA bundle injected by OpenShift for client certificate authentication
3. **Authorize Requests**: Using static authorization config that only allows the `prometheus-k8s` service account to access `/metrics`

## Components

### ServiceMonitor (`monitor.yaml`)

Configures Platform Prometheus to scrape metrics from the operator:
- Uses `scrapeClass: tls-client-certificate-auth` for client certificate authentication
- No bearer token required (unlike User Workload Prometheus)

### CA Bundle ConfigMap (`ca-configmap.yaml`)

A ConfigMap with the `service.beta.openshift.io/inject-cabundle: "true"` annotation. OpenShift's service-ca-operator automatically populates it with the cluster's service CA certificate. This CA is used by Prometheus to verify the metrics endpoint's server certificate.

### Metrics Client CA ConfigMap (`metrics-client-ca-configmap.yaml`)

A ConfigMap containing the client CA certificate used by kube-rbac-proxy to verify Prometheus's client certificate. It starts with a placeholder certificate and is synced by the MetricsClientCA controller from `kube-system/extension-apiserver-authentication`.

### kube-rbac-proxy Config Secret (`kube-rbac-proxy-config-secret.yaml`)

Static authorization configuration that restricts `/metrics` access to only the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account.

### Prometheus RBAC (`prometheus-k8s-role.yaml`, `prometheus-k8s-rolebinding.yaml`)

Namespace-scoped Role and RoleBinding that allow the `prometheus-k8s` service account to discover services, endpoints, pods, and endpointslices in the operator namespace.

## Automatic Setup

When installing the operator via OLM, the namespace will be created with the `openshift.io/cluster-monitoring: "true"` label, which allows Platform Prometheus to discover ServiceMonitors in this namespace.

All required resources (ConfigMaps, Secrets, RBAC) are automatically created as part of the bundle.

## Manual Setup (Development)

If deploying manually without OLM, ensure the namespace has the cluster-monitoring label:

```bash
kubectl label namespace openshift-workload-availability openshift.io/cluster-monitoring=true --overwrite
```

## Verifying Metrics Collection

1. Check that the ServiceMonitor exists:
   ```bash
   kubectl get servicemonitor -n openshift-workload-availability
   ```

2. Check Platform Prometheus targets:
   ```bash
   oc port-forward -n openshift-monitoring prometheus-k8s-0 9090 &
   curl -s http://localhost:9090/api/v1/targets | jq '.data.activeTargets[] | select(.labels.job | contains("node-healthcheck"))'
   ```

3. Query metrics:
   ```bash
   curl -s 'http://localhost:9090/api/v1/query?query=up{job=~".*node-healthcheck.*"}' | jq
   ```

## Troubleshooting

### Target shows DOWN in Prometheus

Check kube-rbac-proxy logs:
```bash
kubectl logs -n openshift-workload-availability deployment/node-healthcheck-controller-manager -c kube-rbac-proxy
```

Verify the client CA is properly mounted:
```bash
kubectl get configmap node-healthcheck-metrics-client-ca -n openshift-workload-availability -o yaml
```

### ServiceMonitor not discovered

Verify the namespace label:
```bash
kubectl get ns openshift-workload-availability --show-labels | grep cluster-monitoring
```
